# Voice Rubric — PRD 语气规格

生成 PRD 正文时，**全程遵循本规格**；初稿完成后用 §5 自检清单逐条过一遍。
目标不是模仿某个具体范例，而是稳定复现下面这套可度量的风格。

> 本文所有示例均为**虚构的「会员订阅」功能**合成例句，仅用于演示风格，不指向任何真实产品。

## 1. 风格画像（一句话）

工程精确 + 产品决策可追溯 + 用户文案有温度的中文 PRD。规则段冷静严谨，用户文案段第一人称有人味；条件—响应句式为骨，带时区、带算例、带异常流为筋。

## 2. 七维风格规格

| 维度 | 规格 | 合成示例（虚构「会员订阅」） |
|------|------|------|
| **句长 / 句式** | 中长复合句，以「若 / 当 / 否则 / 则」组织 if-then 条件句 | 「若用户在续费宽限期内完成支付，则会员权益不中断，权益到期日按原到期日顺延一个计费周期。」 |
| **正式度 / 语域** | 双语域：规则段中性工程语域；用户文案段第一人称、有共情 | 规则：「订阅到期后，付费专属内容入口置灰，免费内容不受影响。」<br>文案：「我们知道，每月这笔订阅是您对我们的信任，我们会努力让它更值得。」 |
| **决策痕迹** | 写出「为什么 / 取舍 / 兜底 / 本期不做」，不只写「是什么」 | 「本期续费提醒只做站内信，不做短信——短信成本高且打开率低，待付费转化模型验证后再评估是否补充。」 |
| **举例方式** | 公式写成可执行形式，后接代入数字的算例 | 「按比例退款：退款金额 = 月费 × (剩余天数 / 30)。例：月费 ¥30，剩余 10 天，则退 30 × 10/30 = ¥10。」 |
| **段落节奏** | 总—分—收：先一句总述策略，分点展开，以「原则 / 注」收口 | 「本次围绕『降低流失、提升续费』两条主线展开：① 增加到期前提醒；② 提供年付折扣。原则：让用户在任何节点都清楚自己的权益状态。」 |
| **术语习惯** | 字段 `snake_case` ＋中文释义；UI 文案用「」，按钮用【】；时间带时区；优先级 P0/P1/P2 | 「字段 `next_renew_time`（下次续费时间，北京时间 UTC+8）。点击【立即续费】弹出收银台，顶部提示「您的会员将于 3 天后到期」。优先级 P0。」 |
| **章节结构** | 修订记录→背景→目标→需求范围→设计稿地址→需求明细（每功能：字段说明 / 交互逻辑 / 权限控制 / 异常流处理）→非功能（退款 / 迁移）→埋点 | 见 §4 骨架模板 |

## 3. 正负例（合成）

**正例（要学）**
1. 决策可追溯：「年付折扣定为 8 折——参考同类产品 7.5~8.5 折区间，取保守值优先保毛利，上线后看转化再调。」
2. 算例落地：公式后必跟代入数字的例子（见上表退款示例）。
3. 异常流穷举：每个功能补「网络异常 / 接口超时 / 重复提交」等失败态及对应提示文案。
4. 用户文案有人味：面向用户的弹窗 / 信件用第一人称「我们」，可致歉、可共情，不堆砌官腔。

**负例（要避免）**
1. 空壳占位：不要留「风险评估：略」这类只有标题没内容的章节——有内容才写，没有就删。
2. 信息挤压：多字段对照一律用规整 Markdown 表格，别把多列信息糊成一段。
3. 模板腔复读：「数据需云端同步成功才生效」这类全局共性，抽到「全局说明」写一次，别每个功能重抄。
4. 残句 / 断字段：保证句子完整、字段名完整，不出现截断。

## 4. 骨架模板

```
一、修订记录            （版本号 / 时间 / 变更人 / 变更内容含动机）
二、背景信息            （解决什么问题，尽量给数据化现状）
三、产品目标            （用户是谁 / 目标 / 要求）
四、需求范围            （功能模块 × 功能点 × 优先级 P0-P2 × 状态[新增/变更]）
五、设计稿 & 交互地址    （原型 / Figma 链接）
六、需求明细
   └ 每个功能点：
       功能描述
       业务规则（if-then 条件句，公式带算例）
       说明
         1. 字段 / 元素说明（snake_case + 中文释义）
         2. 交互逻辑
         3. 权限控制
         4. 异常流处理（场景 → 表现 → 交互）
七、非功能需求          （退款 / 数据迁移 / 性能 / 监控，有则写）
八、埋点               （一级 / 二级 / 三级模块 · 事件名 · 关联变量 · 事件类型 · 触发逻辑）
全局说明              （单位换算 / 时区 / 同步 / 权限等共性规则，抽出只写一次）
```

## 5. 风格自检清单（生成后逐条过）

- [ ] 修订记录在最前，且变更内容写了「为什么」
- [ ] 每条业务规则是 if-then 句
- [ ] 每个公式后都有代入数字的算例
- [ ] 每个功能有异常流（至少网络 / 超时一类）
- [ ] 时间都带时区
- [ ] 字段名 snake_case + 中文释义
- [ ] 没有「略」这种空壳标题
- [ ] 共性规则没有逐功能复读
- [ ] 用户文案段（如有）是第一人称、有温度
