# Content Pipeline — 五步写前管线

PRD 正文不是一次性写出来的。本管线用「先研究、再搭骨架、窄上下文分节起草、跨模型自批、最后整篇统一」五步，把一份能直接落地飞书的 PRD 写出来。

这是 skill 的**内容层**。它的唯一产物是一份 **delivery manifest**（结构化的章节 + 块清单），交给 `delivery/` 引擎渲染进飞书 docx。本管线只管「写得对、写得像人」，不碰飞书 API。

## 0. 三条贯穿全程的纪律

1. **窄上下文起草**：起草某一节时，只喂该节的 brief + voice 规格，不把整篇历史塞进去——上下文越窄越不跑题、越不串味。
2. **跨模型自批**：第 4 步的批判者**必须换一个与起草不同的模型**。同一个模型既当作者又当审稿，发现不了自己的问题；换个模型批，是整条管线质量增量最大的一步。
3. **剥脚手架**：研究笔记、证据表、brief、CoVe 问答、`本节主张：`、`（草稿）`、`[来源1]` 这类**生成痕迹绝不进成品**（见 §4）。决策痕迹（为什么这么做、否决了什么方案）要留——那是好内容，不是脚手架。

中间产物（证据表 / 大纲 / brief / 各节草稿 / 验证记录）都落在 workspace 临时目录，按文件流转，不靠对话累积。

## 1. 五步管线

### Step 1 — 研究 / 取证

多视角向需求材料发问，逐条建立「主张 → 证据 → 来源」：

- 视角清单（按需启用）：**产品**（用户是谁、解决什么、成功长什么样）、**工程**（数据从哪来、边界条件、并发/失败态）、**设计**（关键交互、状态流转）、**法务/合规**（隐私、资金、协议）、**运维**（监控、回滚、降级）。
- 每条主张都要落到具体证据（来自用户给的材料 / 检索）；查不到就标「待确认」，不臆造。
- 产物：一张证据表（主张 / 证据 / 来源），留 workspace。

> 取证可路由到一个「逻辑严、结构化抽取强」的模型（见 §2）。若不可用，本步由当前模型直接做。

### Step 2 — 大纲 + 分节 brief

先搭整篇骨架，再为每个 H2 写一份起草说明书：

- 骨架默认采用 voice rubric §4 的章节结构（修订记录→背景→目标→需求范围→设计稿→需求明细→非功能→埋点），按本次需求增删。
- 每个 H2 的 **brief** 包含：本节**主张**一句话、**3-5 个必须覆盖的证据点**、与上一节的**过渡句**、**目标字数**。
- 产物：outline + 各节 brief，留 workspace。

### Step 3 — 分节起草

逐节起草，**一次只做一节**：

- 输入只给：该节 brief + `voice-rubric.md` 的风格规格。**不要**把整篇已写内容塞进来。
- 严格按 voice rubric 行文：业务规则写成 if-then 句、公式带代入算例、时间带时区、字段 `snake_case`＋中文释义、异常流穷举。
- 中文行文以 Claude（Sonnet 为主，难节用 Opus）起草更稳；见 §2。
- 产物：各节草稿，留 workspace。

### Step 4 — 跨模型自批（CoVe）

对每一节做 Chain-of-Verification，**批判者必须 ≠ 起草者**（见 §2 / §3）：

1. 针对该节草稿生成 **6 个验证问题**（覆盖逻辑自洽 / 事实准确 / 边界与异常 / voice 合规）。
2. 逐题**引证回答**（回到证据表 / 材料，不空答）。
3. 把回答暴露出的失败点**改写**进草稿。
4. 若批判模型不可用且无可替代：该节标记「自批未完成，待人工复核」，**不静默跳过**。
- 产物：每节的验证记录 + 改写稿，留 workspace。

### Step 5 — 拼接 + voice 统一 + 剥脚手架 → manifest

**由单一模型（建议 Opus）整篇过一遍**，防止分节起草带来的风格碎片化：

- 修过渡、去重复、统一术语与语气（对照 voice rubric 全篇校准）。
- 按 §4 **剥掉所有脚手架**；决策痕迹保留。
- 跑 §4 的 scrub 验收 + voice rubric §5 自检清单。
- 把成稿组织成 **delivery manifest**（章节树 + 每节的段落 / 列表 / 表格 / callout / 图片 / mermaid 块），交给 `delivery/` 引擎。标题**不要自己编号**——编号由引擎确定性生成（见 delivery 层 D14）。

## 2. 多模型路由

各步对模型能力的要求不同，按下表分工。**核心约束：第 4 步批判者必须与第 3 步起草者不同源。**

| 步 | 角色 | 建议模型 | 理由 |
|---|---|---|---|
| 1 取证 | 结构化抽取 | 逻辑严的模型（如 GPT 系 codex CLI），不可用则当前模型 | 抽取要严谨、少幻觉 |
| 2 大纲 | 架构 | Opus | 整篇结构判断 |
| 3 起草 | 中文行文 | Sonnet 为主，难节 Opus | 中文表达 Claude 强 |
| 4 自批 | 跨模型批判 | **必须 ≠ 起草模型**（GPT 系优先，否则换另一 Claude 模型） | 跨模型批判 = 最大质量增量 |
| 5 统一 | 整篇收口 | 单一模型（Opus） | 防风格碎片化 |

**外部模型调用合约（若启用 GPT 系等外部 CLI）：**
- **粒度**：取证 / 自批以**单节**为单位调用，不整篇喂——降超时概率。
- **超时与重试**：设合理超时；失败重试 1 次。
- **降级保 author≠critic**：外部模型持续不可用时，批判者改用一个**与起草不同的 Claude 模型**（如 Sonnet 起草 → Opus 批判）。不为了凑齐而引入未纳入本期的模型。
- **输入上限**：单节 brief+草稿控制在合理 token 内，超长先拆子节。

> 具体的本地 CLI 调用方式属部署细节，不写进本公开 skill；按你本机的多模型工具自行接线即可。关键是守住「author≠critic」与「单节粒度」。

## 3. CoVe 自批问题模板

每节生成 6 题，覆盖四类。示例骨架（按本节内容填空）：

1. **逻辑**：本节的业务规则之间有没有互相矛盾或覆盖不全的输入组合？
2. **逻辑**：每条 if-then 规则的「否则 / 异常分支」都写了吗？
3. **事实**：本节引用的数字 / 字段 / 接口行为，能在证据表或材料里找到出处吗？
4. **边界**：极值、空值、并发、重复提交、网络异常 / 超时这些失败态覆盖了吗？
5. **voice**：公式有没有代入算例？时间带时区了吗？字段是 `snake_case`＋中文释义吗？有没有「略」这种空壳？
6. **范围**：本节有没有越界写了别节的内容、或漏掉 brief 要求的证据点？

逐题引证回答 → 把暴露的问题改回草稿。

## 4. 不暴露管线（剥脚手架）

**剥掉（= 生成脚手架，绝不进成品）：** 研究笔记、证据表、各节 brief、CoVe 问答、`本节主张：`、`（草稿）`、`[来源1]` 这类行内来源标记、`本段旨在……` 这类元叙述。

**保留（= 决策痕迹，是好内容）：** 「因为……所以我们假设……」、被否决的备选方案及其理由、「本期不做 X，原因是 Y」。

**scrub 验收（交付前必过）：**
- 脚手架关键词 / 模式扫描（上面那串词一个都不许出现在成品里）。
- 术语表一致性（同一概念全篇用同一个词与同一个 `snake_case` 字段名）。
- 跨章节抽 ≥2 段做风格一致性比对。
- 来源若要保留，只进**附录**，且必须是可点开的飞书链接 / 可达 URL，**禁止**本地文件名。

## 5. voice 锚定与校准

起草与统一全程对照 `voice-rubric.md`。它给的是**可度量的风格规格**（七维 + 正负例 + 自检清单），不是要你模仿某个范例。

需要更贴近某位作者的语气时，跑一轮轻量校准：用 2-3 个 voice 变体写**同一节** → 让作者本人挑出最合意的 / 批注 → 把偏好固化进规格。最少 2 段正例 + 2 段负例即可起步，不必等「完美范例」。

## 6. 规模化编排（多章 PRD 的并行生产线）

§1-§5 的五步管线管「**一节怎么写好**」。当一份 PRD 体量去到十几章 / 几百个 section 时，光靠串行五步走不通——需要把若干「五步管线实例」**并行**起来，并在外层加一层编排纪律才收得了口。本节管「**几十节怎么并行生产且收得了口**」。

整体形态：

```
[反写基线]                       ← 6.1
   ↓
[批量拍板表 → 简报 §「关键拍板」]  ← 6.2
   ↓
[共用简报（单一事实源）]          ← 6.3
   ↓
[逐章 spec × N]                  ← 6.4
   ↓
[N 个子 agent 并行五步管线]      ← §1-§5（每章一个实例，输出 PrdSection[]）
   ↓
[编排者全量校验 → 汇总问题清单]  ← 6.5
   ↓
[单一修复波（一个 agent 一次改完）]
   ↓
[gen-manifest → validate → deliver → readback]   ← delivery/ 层 + delivery/tools/
```

### 6.1 基线先行（W0）

正文动笔前，先把本轮所有新拍板 / 翻案 / 删并改的规则**反写进权威基线**（业务规则基线 + 决策记录），原文带时间戳归档。**正文 subagent 必读的事实源永远是基线 + 设计稿 / 原型代码**，不是「上一轮 PRD」。

没有这一步，每个子 agent 各自从过时材料里抓事实，整本 PRD 必然自相矛盾，改完再改、永远收不了口。详见 `writing-rules.md` §9。

### 6.2 先拍板后动手

动手前把**所有待决问题批量列成一张拍板表**，每题给一个默认建议，一次性问完。拍板结果直接写进共用简报「关键拍板」段（见 §6.3 第 5 槽）；子 agent 写正文时**错引拍板 = 写错**。

不要「写到哪儿发现问题再问一句」。零碎问题打断节奏、决策无台账、子 agent 看到的简报中途变更——这三件事任意一件出现，整批并行任务就要重跑。

### 6.3 共用简报 = 单一事实源

**所有内容子 agent 读同一份简报**，章号差异 / 截图差异 / 字数预算差异由「逐章 spec」（§6.4）承载，简报本身全章节通用。共用简报骨架见 `templates/briefing-template.md`，九段固定，按填空写。

**核心分工原则——方法引用 skill 文件，项目事实内联简报：**

| 内容 | 落点 | 理由 |
|---|---|---|
| 文风规范 / 四块结构 / 差异式 / 图代文 / 一条一事 / 过程清零 / 字数纪律 | **引用 `content/writing-rules.md` §N 与 `content/voice-rubric.md`**（节号点名引） | 方法是一份就够，重复维护会漂移 |
| manifest schema / block 类型规范 | **引用 `delivery/src/lib/manifest.ts` 为权威 + 内联一份最小可抄示例** | schema 是合约，给一份示例方便子 agent 直接照抄 |
| 关键拍板（本轮新决策 + 沿用决策） | **内联在简报「§5 关键拍板」段** | 是项目事实、决定子 agent 写什么 |
| 范围线黑名单（一期 / 二期 / 已删 / 死代码幻影） | **内联在简报「§6 黑名单」段，每条带归属** | 是项目事实，且要作为 verify 配置的源 |
| 必读源文件路径（基线 / 决策记录 / 屏代码 / 截图清单） | **内联在简报「§4 必读源」段** | 是项目路径 |
| 字数预算 / 范例 | 总目标在简报、逐章数字在 spec | 节级精度由逐章 spec 给 |

**WHY**：方法 + 事实分离后，下次项目要换的是简报里的填空槽，不是写法规范本身——这是「省 token」的主要来源。

### 6.4 逐章 spec：机械生成

每章一份起草说明书（`briefs/<ver>/chapter-NN-spec.md`），骨架见 `templates/chapter-spec-template.md`。一份合格的 spec 包含五项：

1. **字数预算**：本章正文字符数硬上限。
2. **截图授权表**：该章允许引用哪些图（屏 id → 文件名 → 章内位置 → 是否 hero）。子 agent 不得自造图名。
3. **改写点矩阵**：保留什么 / 改什么 / 删什么 / 新增什么。逐条列、可勾选。
4. **保留 anchorKey 清单**：编排者后续要在这些锚点 splice 流程图、跨章引用要寻址——子 agent 必须原样保留。
5. **输出文件路径 + 自检要求**：交付到哪个 JSON、写完跑什么自检命令、报告里要贴什么结果。

spec 是「机械」的：除了「改写点」段需要人脑判断，其余四项都能从基线 + 截图清单 + 上一版字数表里**脚本生成**（见 `delivery/tools/`）。

### 6.5 N 并行 + 单一修复波

**起草阶段（并行）**：N 个子 agent 同时各写一章，**直接产出裸 `PrdSection[]`**（路径 `sections/<ver>/chapter-NN.sections.json`，UTF-8、`ensure_ascii=False`、可被 `json.load` 解析）。子 agent 之间不共享上下文、不互相读对方草稿——靠共用简报 + 各自 spec 保证一致性。

**修复阶段（串行，单波）**：编排者**全量校验所有章**后，**把所有问题汇总成一张清单交给一个修复 agent 一次改完**。

> ⚠ **绝不发现一个补一个**：补丁式修复每轮都会暴露新问题——这一轮补的引用、术语、写死文案，下一轮校验时又会牵出别的不一致；永远收不了口（项目实测每晚 100+ 次返工的根因即此）。
> 单一修复波的纪律是：**校验先全量跑完、问题先全量汇总、修复 agent 拿到完整 issue list 再动手**，一次性把所有不一致改齐。

修复 agent 的输入：① 完整 issue list（按章组织、每条带「文件 / 位置 / 现状 / 期望 / 依据」五字段）；② 共用简报（不变）；③ 各章原稿。它**不重写、只修补**，且改完同样要走 §6.6 编排者校验，不能由它自己宣称通过。

### 6.6 编排者纪律

编排者**只做规划 / 拼装 / 校验 / 交付 / 审计，不写一线正文**。一线起草、修复都 delegate 给子 agent；编排者用文件流转和工具校验把它们串起来。

**铁律 · 不信任任何子 agent 的「已自检通过」声明**。子 agent 报告「JSON 已写 / grep 已查 / 字数已达标」时，编排者必须用独立脚本复核：

| 子 agent 声明 | 编排者怎么核 |
|---|---|
| 「已写到 chapter-NN.sections.json」 | `ls -la` 看文件存在 + 字节数 + `python3 -c "import json;json.load(...)"` 解析通过 |
| 「过程产物清零 grep 0 命中」 | 独立跑 `delivery/tools/verify.py`（黑名单/产物从 config 读，不依赖子 agent 自检脚本） |
| 「字数 ≤ 预算」 | 跑 `delivery/tools/wordcount.py` 统一口径复测 |
| 「截图引用合法」 | `verify.py` 屏覆盖 + 截图引用两项 |
| 「manifest 校验通过」 | `delivery/src` 引擎 validate（含 LEVEL JUMP 标题层级检查） |
| 「飞书文档已交付」 | `delivery/tools/readback.py` 直方图比对（headings / grid / image / callout / table 六项 vs manifest 预期） |

工具箱（`delivery/tools/README.md`）就是为编排者纪律存在的——`gen-manifest → validate → verify → deliver → readback` 一条龙，每一步都能独立复核，不依赖子 agent 自己说「我跑过了」。

### 6.7 工作副本隔离

修订**不动线上原 doc**：新 doc 交付、原 doc 留对照；逐章 changelog（改了什么 / 为什么改 / 依据哪条拍板）；结构守恒 diff（改前后 section 数 / anchorKey 对账，确保没丢节没丢锚）。

新 doc 验收通过、需求方确认后再决定是否替换原 doc——替换前永远保留两份对照入口，否则一旦改坏，回不去。

### 6.8 中间产物目录约定（建议）

```
workspace/
  briefs/<ver>/
    common.md                      # 共用简报实例（templates/briefing-template.md 填好）
    chapter-NN-spec.md             # 逐章 spec（templates/chapter-spec-template.md 填好）× N
  sections/<ver>/
    chapter-NN.sections.json       # 子 agent 产出的裸 PrdSection[] × N
  manifests/
    manifest-<ver>.json            # gen-manifest 拼装产物
  screenshots-<ver>/
    *.png                          # grid 引用的截图（与截图清单 JSON 对账）
  reports/
    issue-list-<ver>.md            # 全量校验后给修复 agent 的清单
    readback-<ver>.txt             # API 回读直方图结果
```

`<ver>` = 本轮版本号（如 `v43`）。多版本平行存在便于回滚、对账、做版本间 diff。
