# Writing Rules — 写法规范

`voice-rubric.md` 管「语气长什么样」，本文管「每一类内容怎么写到位」。两者一起用。
核心一句话：**像 PM 在给工程师 / 设计师讲产品，不是 AI 在填模板。** 每段都应能从头自然读到尾。

## 1. 功能描述：自然段叙事，不要伪代码

每个功能模块按下面的顺序展开，但**不要加刚性小标题**（别写「用户体验描述」「触发规则」这种标签），用自然段落把它们串起来。

1. **自然语言叙事（开头）**：2-3 段讲清这个功能是什么、用户在什么场景看到什么、为什么有价值。像跟同事讲故事。
   - ✅ 「当用户在续费宽限期内完成支付，顶部那条到期提示会立刻收起，权益状态回到『已续费』，不再打扰。」
   - ❌ 「用户体验描述：当 `renew.status==paid && in_grace_period` 时……」（伪代码堆条件）
   - ❌ 决策表：条件 → 场景 A → 场景 B → 场景 C
2. **交互状态表**：一张表覆盖所有状态，是功能的「全景图」。

   | 状态 | 用户看到什么 |
   |------|------------|
   | 初始态 | … |
   | 触发（正常） | … |
   | 触发（变体） | … |
   | 空状态 | … |
   | 异常 / 降级 | … |
3. **计算示例**：用行文把账算给人看，**给正例 + 反例**。
   - ✅ 「月费 ¥30，剩余 10 天，按比例退款 = 30 × 10/30 = ¥10；若剩余 0 天，则退款 = ¥0，提示「本周期已无可退余额」。」
   - ❌ 「Step 1 GUARD 检查 → Step 2 TRIGGER 检查 → Step 3 …」
4. **视觉 / 交互参数**：表格列具体值（位置、颜色 hex、动画时长、文案）。多端有差异的分列。
5. **设计理由**：自然穿插或末尾「为什么这么设计」，**不要**独立的 `Rationale` 标签。
6. **验收参考**：自然语言描述测试场景（正常 / 异常 / 降级），**不要** Given/When/Then 代码块。
7. **泳道图（重点功能）**：多角色 / 多系统交接的功能配泳道图，泳道 3-6 条，超出则拆。图怎么落见 §5。

## 2. 后台配置：字段表

涉及后台可配置项，用统一字段表（参考 L&Zone 风格）：

```
**所属后台页面**：[页面路径]

| 字段标题 | 字段名 | 必填 | 配置交互 | 需求描述 | 默认值 | 操作 |
|---------|--------|------|---------|---------|--------|------|
```

每个配置项必须覆盖：**默认值、取值范围、生效时机、校验规则**。字段名用 `snake_case`＋中文释义。

## 3. 埋点独立成章

埋点**单独成章**，不要散落在各功能里。事件表至少含：一级 / 二级 / 三级模块 · 事件名 · 关联变量 · 事件类型 · 触发逻辑。数据接口用正式 JSON Schema 描述。

## 4. Out of Scope 必写

需求范围里**明确写出本期不做什么**及原因。「本期不做 X，因为 Y，待 Z 验证后再评估」——这既是边界，也是决策痕迹（voice rubric 看重的指纹）。

## 5. 图表：写 mermaid 源，渲染交给交付层

**飞书云文档不渲染 mermaid**，所以图表要预生成 PNG 再插入。在内容层，你只负责**写对 mermaid 源码**，把它放进 manifest 的 mermaid 块；渲染成 PNG 并插入飞书由 `delivery/` 层完成，不要在这里手搓渲染命令。常用图的 `.mmd` 起手模板见 skill 的 `templates/` 目录。

**一句话决策表（流程 / 时序 / 状态 / 漏斗类一律 Mermaid，绝不用 matplotlib 手画）：**

| 你要画的是… | 用什么 |
|---|---|
| 时序图（多角色交互、并行、消息往返） | Mermaid `sequenceDiagram` |
| 流程图、决策树、Onboarding 步骤 | Mermaid `flowchart TD` |
| 漏斗、转化阶段、分段分组 | Mermaid `flowchart TD` + `subgraph` |
| 状态机 | Mermaid `stateDiagram-v2` |
| 类图 / ER / 组件关系 | Mermaid `classDiagram` / `erDiagram` |
| 柱 / 饼 / 雷达 / 热力 / 甘特（要数据精度 + 品牌色） | matplotlib |

**Mermaid 陷阱（写源码时少踩这些坑）：**
1. `{` `}` 是菱形保留字符——label 里的 `{name}` 改成 `[name]` 或 `&#123;name&#125;`，否则解析报错。
2. 中文换行用 `<br/>`，不要 `\n`、不要真换行。
3. subgraph 默认横向；长漏斗用 `direction TB` 强制纵向。
4. classDef 着色克制，**全图 ≤3 个 class**——评审场景越克制越专业。
5. `sequenceDiagram` 第一行写 `autonumber`，便于评审讨论「第 5 条消息」。
6. 并行用 `par...and...end`，不要用 Note 模拟。
7. 节点 ID 不能有特殊字符：`S1`/`Start`/`End` 行，`S-1`/`S.1`/`S 1` 报错。
8. label 含引号用 `\"` 转义，或外层用 `["..."]`。

## 6. 写作检查清单（生成后过一遍）

- [ ] 每个功能用自然语言段落开头（不是伪代码 / 决策表）？
- [ ] 读起来像 PM 在讲产品（不是 AI 填模板）？
- [ ] 交互状态覆盖 初始 / 触发 / 空 / 异常？
- [ ] 计算示例有正例 + 反例？
- [ ] 验收参考是自然语言（不是 Given/When/Then）？
- [ ] 后台配置字段表齐全（标题 / 字段名 / 必填 / 交互 / 描述 / 默认值 / 操作）？
- [ ] 埋点是独立章节？
- [ ] Out of Scope 明确？
- [ ] 设计理由自然穿插（不是独立 Rationale 标签）？
- [ ] 流程 / 时序 / 漏斗图用的是 Mermaid（不是 matplotlib 手画）？

## 7.（可选）业务评审方案变体

给老板 / 管理层评审用的方案 ≠ PRD：它卖方向、留选择题，视觉密度高。若用户要的是评审方案而非开发用 PRD，结构换成：行业机会 → 体验路径全景 → 形象 / 呈现菜单（A/B/C/D + 雷达图 + 推荐项，让老板做决定）→ 技术与成本 → **成功判定漏斗**（终局结果 + 过程归因）→ Tier 分级 → 排期里程碑 → 风险与备份。此时「功能描述模板」让位给「图表 + 短文」。默认产出仍是开发用 PRD，除非用户明确要评审方案。
