# Writing Rules — 写法规范

`voice-rubric.md` 管「语气长什么样」，本文管「每一类内容怎么写到位」。两者一起用。核心一句话：**像 PM 在给工程师 / 设计师讲产品，不是 AI 在填模板。** 每段都应能从头自然读到尾；规则自立成文，读者不需要了解任何内部讨论就能执行。

> 本文示例均为**虚构的「云笔记同步设置」功能**合成例句，仅用于演示写法，不指向任何真实产品。

---

## 1. 四块结构 + 职责互斥

每个功能模块固定四块：**功能描述 → 业务规则 → 说明（业务·元素·交互·异常）→ 权限控制**。四块互斥，越界即重写。

| 块 | 写什么 | 不写什么 |
|---|---|---|
| 功能描述（H3） | 这个模块**解决什么问题、是什么、在产品里干什么**。一段总述即够。 | 立项叙事、背景推导、「为了……我们……」二段式铺陈。 |
| 业务规则（H3） | 行为规则 / 约束 / 计算口径 / 状态流转条件 / 权限边界。粗体标签领起 + 完整规则句。 | 界面元素构成（→元素）、状态映射渲染（→状态机图）、控件跳转（→交互）。 |
| 说明 · 业务（H4，ordered） | 该状态下的进入条件、业务语义、计算口径。 | 重复业务规则块已写过的全局规则。 |
| 说明 · 元素（H4，unordered） | 图上看不出的事：数据来源、动态文案、条件露出、业务语义色。 | 居中 / 整宽 / 位置 / 大小等视觉描述（唯一例外：颜色承载业务语义，如错误红）。 |
| 说明 · 交互（H4，ordered） | 「用户做什么 → 系统怎么响应 → 结果（目标屏 / toast 原文）」。 | 与业务规则重复的口径。 |
| 说明 · 异常（H4，unordered） | 「场景 → 表现 → 系统/用户处理 → toast 原文」。 | 已在交互里写过的正常路径。 |
| 权限控制（H3，页面级） | 不同角色 / 端别的可见性与操作权差异。模块所有状态之后统一写一次。 | 逐状态重复权限规则。 |

**WHY**：四块职责互斥避免源头复述。功能描述先回答「解决什么问题」而不是一刀切只写定位，让读者第一眼抓住价值。元素去视觉是因为开发对着 UI 设计稿做，UI 由设计师再设计、必然与原型像素细节不同——元素块写规则而非外观。

**功能描述写什么的判断**：状态展示型入口 = 一句定位 + 功能清单；多步业务流程 / 跨端协作 = 把"解决什么问题、大致怎么运作"讲透。**别一刀切**。

## 2. 差异式逐态写法（压缩主杠杆）

多状态屏不再为每个状态重复全四块。**基准态写全，其余态只写差异。**

1. **选基准态**：屏的首个 / 最常见态。基准态写全四块（业务 ordered / 元素 unordered / 交互 ordered / 异常 unordered）。
2. **其余态只写「与基准态的差异」**：固定开头一句「与「基准态名」的差异：」，然后 2-6 行逐条列差异（元素增减、文案变化、交互分支、进入条件）。基准态已写过的内容**绝不复述**。
3. **差异 >6 条**的态可破例写全四块，但要先自问是不是基准态选错了。
4. **覆盖不减**：每个态仍有自己的 section（保证可寻址、不丢屏不丢态）。差异式改的是表达密度，不是覆盖广度。

**WHY**：实测在保留全部业务事实、写死文案、状态边界一条不少的前提下，整体字数可压 30-40%。多状态屏过去 80% 体量是逐态镜像复述。

**虚构范例**（云笔记同步设置页）。**基准态「默认态」写全**：

> **业务**（ordered）
> 1. **进入条件**：用户已开通云端账号且当前网络在线时，进入设置页即进入默认态。
> 2. **同步周期**：默认每 15 分钟轮询一次；用户主动触发可立即同步，全天累计上限 96 次。
>
> **元素**（unordered，只写图上看不出的事）
> - **上次同步时间**：动态展示服务端记录的最近成功同步时间戳。
> - **「立即同步」按钮**：达到全天上限后置灰（见异常）。
>
> **交互**（ordered）
> 1. 点击「立即同步」→ 触发一次同步 → toast「已同步至最新」→ 停留本页。
>
> **异常**（unordered）
> - 全天累计已达 96 次 → 「立即同步」不可用，提示「今日同步次数已达上限，明日 0 点重置」。

**非基准态「离线态」差异式**：

> 与「默认态」的差异：
> - 顶部新增灰条提示「当前离线，下次联网后自动同步」。
> - 「立即同步」按钮置灰；点击无响应。
> - 「上次同步时间」末尾追加「（离线前）」后缀。

## 3. 图代文：元素块只写图上看不出的事

持截图的态（grid 左图右文），「元素」块**只写图上看不出的规则**：数据来源、条件露出、动态文案、业务语义色、按钮的条件禁用。**图上一目了然的静态元素清单不再逐项枚举**（标题、说明文区、按钮位置等）。

无图的态维持文字完整性（按 §1 四块写）。

**WHY**：开发与设计师看到的是图本身，文字再复述布局 = 两份事实、容易漂移。把"图能交代"的让给图，文字留给"图交代不了"的业务语义。

## 4. 一条一事 + 三级分级

正文文字按三级组织：**节标题 → 粗体标签领起 → 子条**。一个编号 / 圆点只承载一个元素 / 一条规则 / 一步交互。分号只在一条规则内部并列时使用。

- ✅ 「**额度限制**：每 12 个月最多暂停 2 次；额度用尽后不可再暂停。」
- ❌ 「**元素**：上次同步时间；立即同步按钮；高级选项区；底部说明文字。」（四个独立元素挤进一句分号串）

**WHY**：分号串是实际评审中被多轮点名"非常不可读"的根因。三级分级让读者第一眼看到标签即知本条说什么，再读子条获取细节。

## 5. 过程产物清零

**原则**：文档读者是运营、法务、技术等自然人，用于开发。读者没参与过任何内部讨论。**任何内部过程信息对读者都是噪音；规则自立成文——一条规则不需要溯源编号即可执行。**

| 类别 | 写时表现 | 处置 |
|---|---|---|
| 决策 / 需求 / 反馈编号 | 句末「（D-NN）」「（R-NN-NN）」「（C-NN）」一类内部编号括注 | **删除括注**。规则句保留。承载跨章指向的，改为「见第 N 章「节名」」。 |
| 原始素材溯源标注 | 「（取自屏代码）」「（屏代码 `xxx.js`）」「方案 A」「方案 B」 | **删除**。文案直接照写。来源标注不进正文。**屏 ID** 等原型可见的定位键保留。 |
| 内部文档引用 | 「（共用简报 §5）」「（执行方案 §x）」「（增量 02）」 | **删除**，规则正文直接陈述。 |
| 计数对账叙述 | 「记 73 屏/198 态…以代码实测为准」 | **删除**。最终计数只在附录出现一次定稿数字。 |
| 评审期历史注记 | 「评审期曾称方案 B」「为已作废的历史口径」 | **删除**。读者没见过旧口径，直接写现行规则。 |
| 推导链叙述 | 「为什么从 8 态合并为 3 态」的版本演化史 | 对读者有价值的业务理由压成一句话；纯演化史删除。 |

**跨章引用一律「见第 N 章「节名」」**，绝不用编号。**WHY**：H2 全局连续编号会随章节增删漂移；手写编号必漂移。引擎在交付时按 ORDER 全局编号，正文写作期间编号永远是"将要变的"。

**自检命令模板**（交付前必跑，替换 `<目录>` 与具体黑名单模式）：

```bash
# 通用模板：把 <模式> 替换成项目实际要查的内部编号 / 历史口径词
grep -nE '<模式1>|<模式2>|<模式3>' <目录>/*.json

# 示例模式（按项目实际定）：
#   决策编号：（D[0-9]+）|（R[0-9]+-[0-9]+）|（C-[0-9]+）
#   溯源标注：取自屏代码|屏代码|共用简报|执行方案
#   历史口径：评审期|作废|待讨论|方案 A|方案 B
```

要求**零命中**才放行。

## 6. 黑名单 + 白名单机制

**范围线黑名单**：写进任务简报，明确「正文绝不出现的词 / 概念 / 屏名 / 历史口径」。每条标归属（二期 / 已出 scope / 已删 / 死代码幻影）。

**误伤白名单**：黑名单 grep 自检脚本里维护"业务正文里合法出现的同形词"白名单（如「**取消**订阅」是合法功能名，不该被「取消」黑名单条目误伤）。两边配套维护。

**WHY**：差异式 + 清零规则会逼出大量改写，纯人眼难以验收；黑白名单 + grep 把验收自动化。机制通用，**名单本身因项目而异**——写进项目 config，不进 skill。

简报黑名单段最小骨架（写进共用简报的「范围线」节，归属列至少覆盖：二期 / 已出 scope / 已删屏 ID / 死代码幻影 / 旧口径或评审期称谓 / 内部编号格式——后者见 §5）。验收脚本侧的白名单即项目 config 的 `allowed` 列表（见 `delivery/tools/README.md`），机制由 skill 提供，条目由项目维护。

## 7. 文风双边界 + 换词表

> 详细的语气规格与正负例见 `voice-rubric.md`。本节给出**与差异式 / 一条一事配套的双边界界定**与换词表。

**目标 = 专业且可读，两者不矛盾。** 同时躲两个雷区：

1. **下界 · 别太 AI 腔 / 公式速记**：不写「顶部行 = 套餐名 + Chip」这种配置表速记；不堆生造缩写；不一句话堆三个 AI 腔术语（「可锚」「净新增主体」这类生造压缩词一律删）。
2. **上界 · 别太口语化 / 大白话**：不写「怎么进来的 / 免得用户以为能用 / 把支付修好 / 点了会怎样」这种聊天体。

| 别写（口语） | 改写（专业书面） |
|---|---|
| 怎么进来的 | 进入条件 / 何时进入此状态 |
| 免得用户以为能用 | 避免用户误判权益范围 |
| 被推着去 X | 被误导走向 X 流程 |
| 点了会怎样 | 点击后的系统响应 |
| 变脸 / 长这样 | 按状态切换呈现 |
| 说了算 | 以服务端判定为准 |
| 把 X 修好 | 完成 X 修复 |
| 现网 | 现状 / 线上 |

**写紧纪律（一步到位写紧，不写松再删）：**
- **禁二段式铺陈**：功能描述一段总述即够，不前置背景推导、不写「为了……我们……」的立项叙事。
- **禁根因外溢**：业务规则只写规则本身 + 一句业务理由；WHY 不展开成段。
- **禁「承载…/用于…」套话层**：元素直接写它是什么、装什么数据文案，不加「该区域承载了……的功能」垫话。
- **公共规则一律引用「全局说明」章**，不在各章重述。
- **写紧 ≠ 减信息**：四块骨架、写死文案、公式、状态边界一条不能少。砍的是废话层和结构性复述，不是事实。

**其它文风铁律：**
- **形容词后必跟可测硬规则**，不写「流畅顺滑沉浸式」空词。
- **文案 / toast / 提示一律写死**（引号写原文，从屏代码 / 设计稿取，不凭记忆编）。
- **术语降噪但不去专业**：渲染态代号（这类英文短横线 ID，如 `pending-payment`）只在状态机图 / 屏目录出现，正文用对应中文业务态名。
- **设计理由自然融入**："为什么这么设计"用自然段落写在该功能描述里，不要独立 `Rationale` 标签。
- **不写 Given/When/Then**：QA 验收用自然语言示例描述（正常 / 异常 / 降级三场景），不用代码块伪代码。
- **算例落地**：公式后必跟代入数字的例子，给正例 + 反例。
   - ✅「月费 $30，剩余 10 天 → 按比例退款 = 30 × 10/30 = $10；若剩余 0 天，则退款 = $0，提示『本周期已无可退余额』。」
   - ❌「Step 1 GUARD 检查 → Step 2 TRIGGER 检查 → Step 3 …」
- **时间带时区、字段 `snake_case` + 中文释义、UI 文案「」、按钮【】**。
- **交互状态表**用一张表覆盖所有状态（初始 / 触发正常 / 触发变体 / 空 / 异常降级），作为功能全景图。

## 8. 字数预算制

**逐章字数预算先定后写**：开写前给每章一个目标字符数（依据 = 该章业务复杂度 + 屏 / 态数 + 截图数）。预算是硬约束。

**统一字数口径**：交付前用同一套统计脚本复测全章。砍的优先级 = **结构性复述 > 套话层 > 推导叙述**；写死文案、公式、状态边界、异常场景一条不能砍。

**WHY**：实测「写松再删」一定删不动（每句话都有人感觉重要）；只有「一步到位写紧」配合预算，才能在不丢业务事实的前提下落地板。同一份 PRD 在不丢规则的前提下从 10 万字压到 6.5 万字，路径就是这个。

简报里给每个子 agent 的预算段落形如「本章目标字符数 ~X,XXX，写完报告统计值」。

## 9. 基线先行（写之前先反写权威基线）

**铁律**：正文必须 against 正确的权威基线写。否则正文自相矛盾，改完再改、永远收不了口。

开写前的「W0 反写」做两件事：

1. **更新业务规则基线**（如 `XX-业务逻辑基线.md`）：把本轮所有新拍板 / 翻案 / 删并改的规则反写进基线，原文带时间戳归档。
2. **写决策记录**：每条新决策一行（编号 / 时间 / 内容 / 动机）。**决策编号不进正文**（§5），只用于内部追溯。

正文 subagent 必读的源头永远是：**基线（权威事实）+ 决策记录（核对，不引用编号）+ 设计稿 / 屏代码（写死文案的真相源）**。

**WHY**：没有基线先行就开正文 = 每个 subagent 各自从过时材料里抓事实 = 整本 PRD 自相矛盾。这是文档任务 >20min 还没产出时第一个要反思的方向。

## 10. 功能描述：自然段叙事，不要伪代码

四块内部不要加刚性小标签（别写「用户体验描述」「触发规则」这类标签），用自然段落串起来。

- ✅ 「当用户在宽限期内完成支付，顶部到期提示立刻收起，权益状态回到『已续费』，不再打扰。」
- ❌ 「用户体验描述：当 `renew.status==paid && in_grace_period` 时……」（伪代码堆条件）
- ❌ 决策表：条件 → 场景 A → 场景 B → 场景 C

多角色 / 多系统交接的功能配**泳道图**（泳道 3-6 条，超出则拆）。多端差异参数用表格分列。

## 11. 后台配置：字段表

涉及后台可配置项，用统一字段表：

```
**所属后台页面**：[页面路径]

| 字段标题 | 字段名 | 必填 | 配置交互 | 需求描述 | 默认值 | 操作 |
|---------|--------|------|---------|---------|--------|------|
```

每个配置项必须覆盖：**默认值、取值范围、生效时机、校验规则**。字段名用 `snake_case` + 中文释义。

---

## 12. 埋点独立成章 · Out of Scope 必写

**埋点单独成章**，不要散落在各功能里。事件表至少含：一级 / 二级 / 三级模块 · 事件名 · 关联变量 · 事件类型 · 触发逻辑。数据接口用正式 JSON Schema 描述。

**Out of Scope 必写**：需求范围里**明确写出本期不做什么**及原因。「本期不做 X，因为 Y，待 Z 验证后再评估」——这既是边界，也是决策痕迹。

**附录集中文末，不按章插入**：备查内容（验收清单 / 规则溯源 / 穷举表 / 竞品定价 / 技术细节 / 判定矩阵）放文档最末。附录按章插在正文间会打断阅读流。**验收标准也移出正文**，验收另成表格文档（给 QA 用）。

## 13. 图表：写源码，渲染交给交付层

**飞书云文档不渲染 mermaid**，图表要预生成 PNG 再插入。内容层只负责**写对源码**，把它放进 manifest 的 mermaid 块；渲染由 `delivery/` 层完成，不要在这里手搓渲染命令。常用图的 `.mmd` 模板见 `templates/`。

**一句话决策表（流程 / 时序 / 状态 / 漏斗类一律 Mermaid，绝不用 matplotlib 手画）：**

| 你要画的是… | 用什么 |
|---|---|
| 时序图（多角色交互、并行、消息往返） | Mermaid `sequenceDiagram` |
| 流程图、决策树、Onboarding 步骤 | Mermaid `flowchart TD` |
| 漏斗、转化阶段、分段分组 | Mermaid `flowchart TD` + `subgraph` |
| 状态机 | Mermaid `stateDiagram-v2` |
| 类图 / ER / 组件关系 | Mermaid `classDiagram` / `erDiagram` |
| 柱 / 饼 / 雷达 / 热力 / 甘特（要数据精度 + 品牌色） | matplotlib |

**WHY**：matplotlib 手画的流程图「花哨而不专业」（圆角渐变箭头）。工程文档标准是 Mermaid。该用什么用什么——状态机用 `stateDiagram`，流程用 `flowchart`，跨角色 / 跨端用泳道（`subgraph`），纯数据用表格，简单逻辑用文字。**不为「丰富」而堆图**。

**Mermaid 陷阱（写源码时少踩这些坑）：**
1. `{` `}` 是菱形保留字符——label 里的 `{name}` 改成 `[name]` 或 `&#123;name&#125;`。
2. 中文换行用 `<br/>`，不要 `\n` 也不要真换行；label 含引号用 `\"` 或外层 `["..."]`。
3. subgraph 默认横向，长漏斗用 `direction TB` 强制纵向；并行用 `par...and...end` 不用 Note 模拟。
4. classDef 着色克制，**全图 ≤3 个 class**；`sequenceDiagram` 第一行写 `autonumber` 便于评审讨论「第 5 条消息」。
5. 节点 ID 不含特殊字符：`S1`/`Start` 可，`S-1`/`S.1`/`S 1` 报错。

## 14. 引用与标识纪律

- **不写读者无法理解的标识符**：设计稿 fileKey、规则编号（如 R-707）读者根本理解不了。除非能跳转且确有跳转必要，否则不写。
- **已有界面**：要么附链接 / 截图，要么文字描述清楚。不能只写一个 fileKey 完事。
- **引用必须有出处**：写「方案 A」之前必须先定义「方案 A 是什么」，不能莫名其妙出现。
- **运营文档对齐**：运营 / 法务文档已确认的信息，PRD 引用确认事实而非悬空描述。
- **占位项标注「由 X 团队提供」**：运营承接页 / 客服话术 / 法务文案等原型只能占位，PRD 标占位，不自造内容。

## 15. 需求完整性纪律

- **需求 100% 实现，不降级不妥协**：不把「全交互」降为「部分交互」；不把「逐行审计」降为「抽样审计」。这是信任问题。
- **必须折中时先讨论**，不能自行决定。
- **原始需求留档作为基准**：后续所有产出对照基准验收。
- **不擅自分期**：需求梳理阶段不引入「第一版 / MVP / 二期 / 后置」。**P0-P3 是轻重缓急，不是版本 / 分期**。
- **纯需求方视角**：不碰表结构 / 字段，只提逻辑要求。不预判现状 / 不标「已有 / 可复用」。开发可能正在重构后台，产品规定表结构会与新框架对不上。
- **表现层细分**：移动 App / 大屏端 / 客服后台等多端按需分列。

## 17.（可选）业务评审方案变体

给老板 / 管理层评审用的方案 ≠ PRD：它卖方向、留选择题，视觉密度高。若用户要的是评审方案而非开发用 PRD，结构换成：行业机会 → 体验路径全景 → 形象 / 呈现菜单（A/B/C/D + 雷达图 + 推荐项，让老板做决定）→ 技术与成本 → **成功判定漏斗**（终局结果 + 过程归因）→ Tier 分级 → 排期里程碑 → 风险与备份。此时「四块结构」让位给「图表 + 短文」。默认产出仍是开发用 PRD，除非用户明确要评审方案。
