## 覆盖要求

- 以底稿为准，底稿覆盖的**每个屏 / 态都要有对应 section 或 grid**；除下方「改写点」明确列删的项外，**一个都不能丢**。
- 「改写点」列明的新增屏 / 态必须补写。
- 从零写章：以共用简报 §4 给的设计稿 / 屏代码为准，覆盖该章所有屏 / 态。

## 改写点（必须全部执行，逐条勾选）

> **本节是 spec 唯一需要人脑判断的部分**——按本轮拍板 / 评审反馈逐条列出本章要保留 / 修改 / 删除 / 新增的事项。每条尽量带「依据」（指向共用简报 §5 拍板编号或基线某章节）。

- [ ] **保留**：`{{本章上一版写对的核心规则，例：m-pause 暂停 30 天固定 + 额度 2 次/12 个月}}`。依据：`{{共用简报 §5 沿用拍板 N / 基线 §X}}`。
- [ ] **改**：`{{要改的口径，例：active-dev 状态卡删「待讨论」句，写明设备家庭版无升级入口}}`。依据：`{{共用简报 §5 新拍板 N}}`。
- [ ] **删**：`{{要删的口径 / 屏 / 态，例：「同步设置」删 协同编辑 / 团队空间 两态全部 section}}`。依据：`{{共用简报 §6 黑名单条目}}`。
- [ ] **新增**：`{{要补写的屏 / 态，例：m-cancel-entry 新增第 4 态 member}}`。依据：`{{共用简报 §5 新拍板 N / §4 必读素材}}`。
- [ ] **机械清理**：本章过程产物清零项 — `{{本章高频出现的内部编号 / 溯源词，例：屏代码溯源 27 处全部删除；D-NN 编号 45 处全部删除}}`。依据：`content/writing-rules.md` §5。
- [ ] **字数压缩**：`{{压缩目标，例：差异式压缩 {{底稿字符数}} → ≤ {{本章字数上限}}}}`；优先级 = 结构性复述 > 套话层 > 推导叙述；写死文案 / 公式 / 状态边界一条不能砍（见 `writing-rules.md` §8）。

> 填写说明：每条改写点必须可验证——子 agent 完成后编排者 grep / diff 一一对账。模糊改写点（如「整体精简」「优化语气」）禁列；要么拆成可勾选条目，要么不列。**从零写章**（无底稿）时本矩阵只保留「新增」「机械清理」「字数压缩」三类，其余各类划掉不填。

## 输出文件路径 + 自检要求

- **落盘路径**：`workspace/sections/{{版本号}}/chapter-NN.sections.json`
- **写入纪律**：分段写入，单次 ≤150 行；先 Write 骨架，再 Edit 追加。
- **解析自检**：`python3 -c "import json;json.load(open('路径'))"` 必须通过。
- **过程产物 grep**（贴出命令与结果，要求零命中；模式从共用简报 §2.5 拼装）：
  ```bash
  grep -nE '{{本项目内部编号 + 溯源词正则，与共用简报 §2.5 对齐}}' sections/{{版本号}}/chapter-NN.sections.json
  ```
- **黑名单 grep**（共用简报 §6 每条逐个 grep，正文零命中）。
- **字数复测**：`python3 delivery/tools/wordcount.py sections/{{版本号}}/chapter-NN.sections.json`，TOTAL ≤ {{本章字数上限}}。
- **覆盖核对**：本 spec 截图授权表 + 改写点新增态 vs 实际 sections 的 anchorKey 列表，差集应为空。
- **报告要求**：交付报告里必须贴出 ① 文件 `ls -la` ② JSON 解析结果 ③ section 数 / 字节数 / 正文字符数 ④ 过程产物 grep 命令与结果 ⑤ 黑名单 grep 结果 ⑥ wordcount.py TOTAL ⑦ 改写点逐条勾选结果。**只有报告完整、编排者复核通过，本章才算交付。**
