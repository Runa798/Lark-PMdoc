## 0. 你要交付什么（输出契约 — 严格遵守）

- 产出**一个 JSON 文件**，路径 = `workspace/sections/{{版本号}}/chapter-NN.sections.json`（NN = 你的两位章号，见你的 spec）。
- 文件内容 = **一个 JSON 数组**，元素是 `PrdSection` 对象（**不是** `{title, sections}` 包裹，就是裸数组 `[ {...}, {...} ]`）。
- UTF-8、`ensure_ascii=False`。写完用 `python3 -c "import json;json.load(open('路径'))"` 自验能解析。
- 写完用 `ls -la` 确认文件存在并报告字节数 + section 数 + 正文字符数。**不要只在总结里声称已写，必须实际落盘并验证。**
- **写文件必须分段**：先 Write 骨架（前几个 section），再逐段 Edit 追加，单次写入 ≤150 行。一次性灌全章会 socket 断连（实测）。
- 文本块用 paragraph / list / table / callout 四种 + grid（见 §1）。**不要用 image / mermaid**（流程图由编排者按 anchorKey 统一插入）。

## 1. 原生 manifest schema（精确，照抄结构）

> **权威**：以 `delivery/src/lib/manifest.ts` 的 `PrdSection` / `Block` 类型为准。本节给最小可抄示例——**常规章节照本节示例即可动笔**，只有用到示例未覆盖的块属性时才需要查 manifest.ts。

顶层是 `PrdSection[]`。每个 `PrdSection`：

```json
{
  "level": 1,
  "title": "购买与支付",
  "anchorKey": "ch2-purchase",
  "blocks": [ /* 至少 1 个 block */ ]
}
```

`level` 整数 1-5（章=1，功能模块=2，块=3，逐状态=4）；`title` 不带前导编号（引擎自动编号）；`anchorKey` 全局唯一稳定 id（英文短横线，前缀章号）。

文本 block 四种：

```json
{ "kind": "paragraph", "text": "正文，可含 **加粗**。一段一意，不含换行符。" }

{ "kind": "list", "list": { "style": "ordered",   "items": ["第一条", "第二条"] } }
{ "kind": "list", "list": { "style": "unordered", "items": ["要点", "要点"] } }

{ "kind": "table", "table": {
    "header": ["列1", "列2"],
    "rows": [ ["a1","a2"], ["b1","b2"] ]
} }

{ "kind": "callout", "callout": {
    "emoji": "pushpin",
    "backgroundColor": 1,
    "borderColor": 1,
    "lines": ["第一行", "第二行"]
} }
```

grid（左图右文；注意 grid 内部 list 是**扁平 style/items，不嵌套 list 包裹**，与普通 list block 不同）：

```json
{ "kind": "grid", "grid": {
    "image": { "path": "screenshots-{{版本号}}/{{屏id}}--{{态}}.png" },
    "widthRatios": [42, 58],
    "blocks": [
      { "kind": "paragraph", "text": "**业务**" },
      { "kind": "list", "style": "ordered",   "items": ["**进入条件**：……"] },
      { "kind": "paragraph", "text": "**交互**" },
      { "kind": "list", "style": "ordered",   "items": ["点击「确认」→ ……"] }
    ]
} }
```

**硬约束（违反会校验失败、交付中断）：**
- `list.items`、`callout.lines`、`paragraph.text`、`table` 单元格**绝不含换行符 `\n`**；要分行就拆多个 item / line / block。
- `table` 每行列数 = header 列数；callout 颜色在范围内；`level` ∈ 1-5；每个 section `blocks` ≥1。
- 内联加粗只用 `**文字**`；不要用 `*斜体*`、`#`、`>` 等其它 markdown。
- grid 的 `image.path` 只能用你 spec 截图授权表里的文件名（前缀 `screenshots-{{版本号}}/`），不得自造；grid 内 ordered list 编号各自重启。
- `anchorKey` 全局唯一，前缀章号（如 `ch2-purchase-start`）。spec 中列出的「必须保留 anchorKey」原样保留——编排者会在这些锚点插流程图或被跨章引用。

## 3. 结构与差异式逐态写法

**详细规则见 `content/writing-rules.md` §1（四块结构）+ §2（差异式逐态）+ §3（图代文）+ §4（一条一事三级分级）。本简报不再复述。**

骨架口诀：**功能描述 → 业务规则 →（逐状态：业务/元素/交互/异常）→ 权限控制**。
- 基准态写全四块；其余态「与基准态的差异：」2-6 行。
- 持图的态用 grid，元素块只写图上看不出的事；无图的态走普通 section 四块写齐。
- 业务规则不写界面元素 / 状态映射 / 控件跳转；元素去视觉。

字数预算由你的 spec 给出（节级精度）。
